# check-rules v2 设计文档：从 Tool 进化为 Plugin

## 背景与问题

check-rules v1 是一个 OpenCode custom tool，用便宜模型（DeepSeek V4 Flash）在代码写完后检查是否违反用户的工程规则。它已被验证有价值（30 天真实使用中被调用 176 次）。

但 v1 有一个 README 自己承认的根本软肋：

> **它依赖模型主动调用。** 便宜模型经常"忘记"在写完代码后调用 check-rules。强约束提示词（"必须调用，不可跳过"）能提高概率，但和所有依赖模型主动选择工具的机制一样，**无法保证**。

README limitations 已经指明了唯一可靠的出路：

> 若要确定性执行（每次写完都检查，不管模型记不记得），`tool.execute.after` plugin hook 是唯一可靠路径。

v2 就是把这条"唯一可靠路径"实现出来：从"求模型记得调用的 tool"进化为"确定性自动触发的 plugin"。

## 一个被否决的错误设计（记录在案，因为它是关键决策）

最直觉的 plugin 设计是：**在每次 `edit`/`write` 之后立即跑规则检查**。这个设计是错的，原因有三：

1. **摧毁心流 / 降低效率。** 开发中大量 edit 是中间态（写一半、调试、改了又改）。每次 edit 都触发检查会不断打断工作节奏。
2. **成本失控。** 真实数据里 edit 有 1334 次。若每次 edit 跑一次 LLM 检查，就是 1334 次 API 调用，绝大多数是对半成品的无效检查。
3. **假阳性泛滥。** 对半成品代码跑规则检查，会报大量"还没写完所以看起来违规"的假违规，噪音淹没真信号。

**结论：不能在"编辑边界"检查，必须在"完成边界"检查。**

## 核心设计：追踪与检查解耦

把"标记哪些文件被改过"和"运行规则检查"两件事解耦到两个不同的 hook、两个不同的时机：

### 1. 编辑边界（轻）：`tool.execute.after` 只做标记

- 监听 `tool.execute.after`，当 tool 是 `edit` / `write` / `patch` 时，把目标文件路径加入"本 session 的脏文件集合"。
- 这一步**零 LLM 调用、零阻断、零延迟**。中间态怎么反复改都无所谓——集合去重，改 100 次同一文件仍只是集合里的一个条目。

### 2. 完成边界（重）：`session.idle` 时批量检查

- 监听通用 `event` hook，筛选出 `session.idle` 事件。
- `session.idle` 的语义（已在 OpenCode 源码 `run-state.ts` 确认）：**agent 完成一轮工作、把控制权交回给用户等待输入的那一刻**。这精确对应"AI 写完代码、还没测试"的理想时机。
- 触发时：取出该 session 的脏文件集合，对每个文件跑一次规则检查，清空集合，把违规结果汇总注入对话。

### 这样解决了什么

- **中间态永远不被检查** —— 检查只在完成态发生。
- **成本从"每次 edit 一次 LLM"降到"每个工作单元一次 LLM"** —— 改 10 次文件 = idle 时检查 1 次。
- **确定性触发** —— 检查由 plugin hook 触发，不再依赖模型"记得"调用。这是相对 v1 tool 的核心进化。
- **不打断、不阻断** —— 只把违规结果注入对话，agent 和用户看到后自行决定是否修复。放弃"违规即阻断 edit"的方案（同样会误伤半成品、且 LLM 检查器有假阳性风险）。

## 触发语义细节

- **只对代码文件标记。** 复用 v1 的 `DOMAIN_MAP`（按扩展名判断），非代码文件（如 `.md`、`.txt`、`.json` 等未映射类型）不加入脏集合，避免无意义检查。
- **每个 session 独立维护脏集合。** 以 sessionID 为 key。不同 session 互不干扰。
- **检查后清空集合。** 一轮 idle 检查完即清空，下一轮工作重新累积。

## 待定参数的决策

1. **脏文件集合的存储：纯内存。**
   - 理由：idle 是短周期事件，脏集合的生命周期就是"一轮工作"。进程重启丢失无所谓——重启后本来就是新一轮工作。不需要持久化，避免引入额外复杂度。

2. **idle 时脏文件很多的限流：并发检查 + 上限。**
   - 理由：agent 一口气改 15 个文件的情况存在。全部串行太慢，全部并发可能打爆 API。采用并发但设上限（如同时最多 5 个），兼顾速度与稳定。

3. **发布形态：并入现有 check-rules repo，作为 v2。**
   - 理由：plugin 和 tool 是同一个东西的两种形态，共享 `detectDomains` / `loadRules` / `checkWithModel` 核心逻辑。README 里那段"limitations 提到 plugin hook 是唯一可靠路径"正好是 v2 的引子，叙事完整。

## 兼容与复用

- **保留 tool 形态作为手动兜底。** 用户仍可手动"检查这个文件"。plugin 自动触发与 tool 手动触发并存，互补。
- **复用 v1 核心逻辑不重写：** 领域检测（`detectDomains`）、规则加载（`loadRules`）、模型检查（`checkWithModel`）三块逻辑抽出为共享模块，tool 和 plugin 都调用它。

## 配置项（沿用 v1 + 新增）

沿用 v1 的环境变量：`DEEPSEEK_API_KEY`、`CHECK_MODEL`、`CHECK_API_BASE`、`ENGINEERING_BRAIN`。

新增（plugin 专属，均有合理默认值）：
- 是否启用自动检查（默认启用）
- 并发检查上限（默认 5）

## 验证方式

- **plugin 能被 OpenCode 加载**：作为 file plugin 在 `opencode.jsonc` 的 `plugin` 数组中引用，启动无报错。
- **脏文件追踪正确**：连续 edit 多个文件后，脏集合包含且仅包含代码文件、去重、非代码文件被排除。
- **idle 触发检查**：agent 完成一轮工作后，观察到对脏文件的批量检查发生、结果注入对话、集合清空。
- **中间态不被检查**：对同一文件多次 edit，确认检查只在 idle 发生一次，而非每次 edit。

## 作品叙事（本设计对求职 proof-of-work 的意义）

这个 v2 的价值不在"又一个检查工具"，而在**它体现的真实工程判断**：

> "我做了个自动代码规则检查插件。第一版想在每次 edit 后检查，但真实使用中发现它摧毁心流、烧钱、对半成品报假阳性。于是我把'脏文件追踪'和'规则检查'解耦——编辑时零成本标记，只在 agent 完成一轮工作（session.idle）时批量检查一次。这既保证了确定性触发（不依赖模型记得调用），又避免了对中间态的无效检查。"

全部来自真实使用，每个决策都能讲清取舍——这正是"深度使用并改进 AI 工具"这一定位的具体证据。
