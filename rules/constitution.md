## 改 bug 前先全局 grep 同类 pattern，一次修完
**场景**:修复某个 bug 或错误模式。
**做**:先 grep 整个项目找到所有同类出现，列清单，一次性全部修完再提交。
**禁**:只修报告的那一处就以为完了。
**原因**:AI 写的同一个 pattern bug 会出现在多个位置。

## 改字段/函数签名/枚举后，审计所有生产者消费者
**场景**:修改 field、function signature、provider、DB column、enum value。
**做**:grep 所有读写位置，确认全部对齐。
**禁**:改完就停，不追查下游引用。

## 单一真相源
**场景**:任何定义、配置、策略、数据。
**做**:只存放在一个权威位置，所有其他文件引用它。
**禁**:复制粘贴到多处；改只改了一处忘了另一处。

## 复杂度螺旋上升时先问能不能砍
**场景**:方案从简单突然需要新技术栈/大重构。
**做**:停下来列出复杂度升级的成本，确认需求本身是否必要，再决定。
**禁**:硬做完一个成本爆炸的方案。

## 公式/单位换算先代入数值手算验证
**场景**:涉及跨精度乘除的公式（如 shares×100 × price×10000 / divisor）。
**做**:代具体数字手算验算，确认结果一致再写代码。
**禁**:把设计文档的公式当真理直接实现。

## 改 UI 前追溯完整数据流
**场景**:修改 UI 的 label、语义、单位、交互。
**做**:grep 全链路：输入 → 存储 → 读取 → 计算 → 展示，列出所有影响点。
**禁**:只改 UI 表面以为就完了。

## DISCUSS FIRST
**场景**:用户提出功能或需求时。
**做**:先讨论方案、取舍、设计。用户明确说「做吧」再写代码。
**禁**:收到需求直接开写。

## 规则:DRY — 先搜索现有实现,禁止重复造轮子

**场景**:写新功能前。

**做**:先浏览项目搜索同名函数/类/工具模块;复用已有 base class、mixin、decorator。

**禁**:不搜索就直接写新函数;复制粘贴相似逻辑。

## 规则:写 bug 修 bug 必须有测试

**场景**:加新功能或修 bug。

**做**:单元测试覆盖正常路径 + 关键边界;修 bug 时先补一个能复现的测试。

**禁**:无测试就声称修完。
